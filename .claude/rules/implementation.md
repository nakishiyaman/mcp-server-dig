# 設計・実装の規律

## 原則: 原因を直せ、症状を消すな

コードを書く前に必ず3段階を踏む。省略しない。

### 1. 原因の特定

修正に着手する前に「なぜそうなっているのか」を言語化する。

- バグ報告 → 再現条件と根本原因を特定してから修正方針を決める
- テスト失敗 → 失敗の原因が「本番コードの不具合」か「テスト環境の不備」かを分類する
- CI/CDの問題 → プラットフォームの仕様・制約を調査してから対策を決める
- パフォーマンス問題 → ボトルネックを計測で特定してから最適化する

「よくわからないが、これを足したら直った」は解決ではない。説明できない修正は入れない。

### 2. 修正のシミュレーション

コードを書く前に「この変更で目的の状態になるか」を頭の中で検証する。

- 入力→処理→出力を具体値でトレースする
- 副作用の影響範囲を確認する（他の機能が壊れないか）
- エッジケースを想定する（空、null、境界値）

### 3. 隠蔽と解決の区別

以下に該当する修正は隠蔽であり、解決ではない:

| やりがちなこと | 何が問題か | 正しいアプローチ |
|--------------|-----------|----------------|
| `try-catch` でエラーを握り潰す | 異常を検知できなくなる | エラーの原因を修正するか、適切にハンドリングする |
| `@ts-ignore` で型エラーを消す | 型安全性が崩れる | 型定義を正しくするか、型ガードを書く |
| `as any` で型を強制する | 型システムを無効化している | unknown + 型ガード、ジェネリクス、正しい型定義で解決する |
| `eslint-disable` で警告を消す | コード品質ルールを無視している | ルールが正しいなら従う、不適切なら設定を変える |
| テストのアサーションを緩める | テストが問題を検出できなくなる | 本番コードを修正してテストを通す |
| `--admin` でブランチ保護をバイパスする | 保護の意味がなくなる | CIが通るようにパイプラインを修正する |
| 手動操作で一時的に回避する | 次回も同じ問題が起きる | 自動化で恒久的に解決する |
| 1箇所の修正を確認せず他に展開する | 間違った修正が増殖する | まず1箇所で正しさを検証してから展開する |

## インフラ・ワークフロー変更の規律

CI/CD、リポジトリ設定、npm公開フローなど、インフラレベルの変更を提案する前に:

1. **現行アーキテクチャを理解する** — `.claude/rules/git-workflow.md` と `.github/workflows/` を必ず読む
2. **制約を調査する** — GitHub Actions、npm、release-pleaseの公式ドキュメントで仕様を確認する。推測で提案しない
3. **恒久的な解決策を設計する** — 場当たり的な回避策は提案しない。「この変更で今後同じ問題が二度と起きないか」を基準にする
4. **影響範囲を列挙する** — Secrets、リポジトリ設定、ワークフロー間の依存関係を確認する
5. **変更後にドキュメントを更新する** — `rules/git-workflow.md` に反映する

## ゴール駆動実装

プランの変更ファイルリストは仮説であり、問題文が真のゴール。

1. **実装の各ステップでゴールに立ち返る** — 「この変更で問題は解決に近づいたか」を都度確認する
2. **完了判定はゴールで行う** — 「リストのファイルを全部変更したか」ではなく「問題が解決されたか」で判定する
3. **リストが不完全と判明したら問題文に戻る** — タスクリストは仮説なので、不足に気づいたら問題文から正しいアクションを導出する

## 設計判断

### 複雑さの導入に慎重になる

- 抽象化は同じパターンが3回出てから検討する（Rule of Three）
- 「将来必要になるかも」で機能を足さない（YAGNI）
- 新しい依存を追加する前に、既存の手段で解決できないか確認する

### 方針と実装の整合性チェック

設計方針を決定した後、各ケースの実装時に「この動作は決定した方針と整合しているか」を明示的に確認する。

- 方針決定後に個別ケースを実装するとき、方針文をそのまま判断基準として使う
- 「目の前の問題が消えるか」ではなく「方針に沿っているか」で実装を評価する
- 方針と矛盾する実装が必要な場合は、方針自体を見直すか、例外として明示的に記録する

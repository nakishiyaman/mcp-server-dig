# テストルール

## TDD規律: Red-Green-Refactor を厳守する

### RED: テストを先に書いて失敗を確認する
- 本番コードを書く前に `.test.ts` を作成する
- `npx vitest run <ファイル>` で失敗を確認する
- 失敗理由を分類する:
  - **環境不備**（missing module, setup等）→ テストセットアップを修正
  - **未実装** → 本番コードを書く
- 環境不備を本番コードの変更で回避してはならない

### GREEN: 最小限のコードでパスさせる
- テストが求める振る舞いだけを実装する
- 「ついでに」の実装をしない

### REFACTOR: テストが緑のまま整理する
- 本番コードとテストコードの両方をリファクタ対象にする

## 絶対に守ること

### 本番コードにテスト都合を持ち込まない
- `skipXxx`、`_testOnly_`、`if (process.env.NODE_ENV === 'test')` は禁止
- テスト環境の問題はテスト環境側で解決する（setup, mock）
- 解決策が本番APIを変更する方向に向かったら、立ち止まって根本原因を再確認する

### テストが失敗したとき、テストを緩めて通す方向に逃げない
- アサーションを弱める（`toEqual` → `toBeDefined`）のは問題の隠蔽
- モックで本物の振る舞いを上書きするのは偽の安心
- 失敗は「何かがおかしい」シグナル。原因を特定して正しく修正する

## mcp-server-dig 固有のテストパターン

### パーサー単体テスト
- サンプルgit出力文字列をテストデータとして使う
- 実際の `git` コマンドは実行しない
- パーサーの入出力を純粋関数として検証する
- ファイル: `src/git/parsers.test.ts`

### ツール統合テスト
- 一時gitリポジトリを `beforeAll` で作成する
- 実際の `git` コマンドでコミット履歴を構築する
- ツールのMCPレスポンス形式（`content[0].text`）を検証する
- `afterAll` で一時ディレクトリをクリーンアップする
- ファイル: `src/tools/tools.integration.test.ts`

### モック方針
- **モック禁止**: git CLI の実行結果（統合テストの価値そのもの）
- **モック対象**: なし（現時点で外部サービス依存がない）

## テストの書き方

- 1テスト1振る舞い
- AAA (Arrange-Act-Assert) パターン
- テスト名は振る舞いを日本語で記述（例: `"ファイルの変更履歴を取得する"`）
- テストファイルはソースファイルと同じディレクトリにコロケーション
